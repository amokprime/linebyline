## Problem Diagnosis

The popup image confirms that the `alert()` **IS being triggered** (the popup shows "No timestamps in main field."). However, the test is timing out because of how Playwright handles native browser dialogs.

### Root Cause

When `keyboard.press('Control+6')` is executed:
1. The keypress triggers `mergeTranslations()` 
2. `alert('No timestamps in main field.')` is called
3. **The native browser alert blocks JavaScript execution**
4. Playwright's `keyboard.press` waits for the page to become responsive
5. Meanwhile, the dialog event fires, but there's a timing issue

The issue is that **Playwright dialogs must be explicitly dismissed** when you capture them via `waitForEvent`. The `Promise.all` pattern captures the dialog object but doesn't auto-dismiss it, causing `keyboard.press` to hang waiting for the page to unblock.

### The Fix

Use `page.on('dialog', handler)` or `page.once('dialog', handler)` which handles the dialog **before** the blocking occurs:

```js
test('warning-no-timestamps', async ({ page, media }) => {
  await page.locator('#file-picker').setInputFiles([
    media('audio.mp3'),
    media('plain_english.lrc'),  // no timestamps
  ]);
  await page.keyboard.press('Control+4');
  const [fileChooser] = await Promise.all([
    page.waitForEvent('filechooser'),
    page.locator('.field-col:not(#main-col) button[title="Open (Middle click)"]').click(),
  ]);
  await fileChooser.setFiles([
    media('plain_spanish.lrc'),
  ]);
  
  // Set up dialog handler BEFORE the action that triggers it
  page.once('dialog', async dialog => {
    expect(dialog.message()).toBe('No timestamps in main field.');
    await dialog.dismiss();
  });
  
  await page.keyboard.press('Control+6');
});
```

Or, if you need the dialog object for assertions later:

```js
// Capture dialog in a variable
let capturedDialog;
page.once('dialog', async dialog => {
  capturedDialog = dialog;
  await dialog.dismiss();
});

await page.keyboard.press('Control+6');
expect(capturedDialog.message()).toBe('No timestamps in main field.');
```

### Why `Promise.all` Doesn't Work Well Here

The `Promise.all([page.waitForEvent('dialog'), page.keyboard.press(...)])` pattern has a race condition:
- `keyboard.press` sends the keystroke and **waits for completion**
- The alert blocks the page
- `waitForEvent` fires but the dialog object isn't auto-dismissed
- `keyboard.press` never resolves because the page is still blocked by the undismissed dialog

Using `page.once('dialog', handler)` ensures the dialog is handled (dismissed) immediately when it appears, unblocking the page and allowing `keyboard.press` to complete.

By the time `keyboard.press` completes, Playwright has already auto-dismissed the dialog, and if your assertion was in that handler, it may or may not have run depending on timing.

## If You Still Want to Catch the Dialog

Use `page.waitForEvent('dialog')` **before** the action, and handle dismissal inside the same promise chain:

```js
const [dialog] = await Promise.all([
  page.waitForEvent('dialog'),
  page.keyboard.press('Control+6'),
]);
expect(dialog.message()).toBe('No timestamps in main field.');
await dialog.dismiss();
```

This pattern ensures the `waitForEvent` is set up before the action fires. But even this can be flaky if Playwright's internal auto-dismiss races with your handler.

## Recommendation

Go with the **outcome-based assertions** (disabled button + unchanged fields). Native `alert()` dialogs are intentionally hard to test in Playwright because they're blocking UI primitives that real browsers are moving away from. Verifying the *effect* (or lack thereof) is both more robust and more meaningful as a test.

## Code Analysis

Looking at `_buildMergedResult` (lines 1615-1627):

```js
function _buildMergedResult(mainLines,tsLines,contentTs,secData){
  const result=[...mainLines];
  for(let ci=contentTs.length-1;ci>=0;ci--){
    const [lineIdx,nextMs]=contentTs[ci];
    if(nextMs===null)continue;
    const ins=[];
    secData.forEach(({lines})=>{if(ci<lines.length)ins.push(lines[ci]);});  // ← iterates ALL secondaries
    const n=ins.length;
    result.splice(lineIdx+1,0,...ins.map((t,j)=>`${msToTs(nextMs-(n-j)*10)} ${t}`));  // ← timestamp staggering depends on n
  }
  return result;
}
```

**Key difference with multiple secondaries:**
- `secData.forEach` collects from ALL secondary fields
- Timestamp staggering `nextMs-(n-j)*10` depends on `n` (number of translations per line)
- There's a guard `if(ci<lines.length)` for when secondaries have unequal line counts
