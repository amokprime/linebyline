Yes, there are several worthwhile unit tests that would complement your existing Playwright QA spec files. Here's my analysis:

## What the existing Playwright specs cover well

Your ~14 spec files exercise **integrated browser behavior** end-to-end: import flows, paste handling, playback controls, mode switching, settings persistence, undo/redo workflows, merge, sync/timestamp, hotkey capture, keyboard navigation, accessibility, and visual regression. These are excellent QA/integration tests — they verify that the whole system works together from the user's perspective.

## What's missing — and where unit tests would add value

The gap is **pure-logic functions** buried inside the single-file HTML that the Playwright specs can only exercise through the full UI. Unit tests would catch regressions in these functions faster, cheaper, and with much better failure diagnostics. Here are the specific candidates, grouped by priority:

### 1. LRC Parsing Functions (highest value)

These are pure functions with zero DOM dependency and complex edge cases:

| Function | Why unit test it |
|---|---|
| `tsToMs(l)` | Corner cases: `[99:59.99]`, `[00:00.00]`, malformed timestamps (`[00:00.9]`, `[0:0.0]`), non-matching lines |
| `msToTs(ms)` | Boundary: `0`, `5999990` (99:59.99), negative input (should clamp to 0), rounding of centiseconds |
| `replaceTs(l, ms)` | Lines with timestamps vs. without, timestamp at exact boundary |
| `isEndTs(l)` | Timestamp-only line, timestamp + whitespace, timestamp + text |
| `normalizeLrcTimestamps(text)` | 3-decimal → 2-decimal truncation, mixed 2/3-decimal, no-op on already-normalized |
| `stripSecLine(l)` | With/without timestamp, leading space handling |

**Why this matters:** Your Playwright specs test these indirectly through import/paste/sync flows, but a bug in `tsToMs` that misparses `[01:60.00]` or a rounding error in `msToTs` would produce a mysterious snapshot mismatch somewhere downstream. A unit test pinpoints it immediately.

### 2. `_peelLastParen` and `batchSplitParens` (high value)

`_peelLastParen` is a non-trivial string algorithm with real edge cases:

| Input | Expected | Why it could regress |
|---|---|---|
| `"hello (world)"` | `["hello", "(world)"]` | Baseline |
| `"a (b (c) d)"` | `["a", "(b (c) d)"]` | Nested parens — must match outermost from end |
| `"no parens"` | `null` | No match |
| `"(only)"` | `["", "(only)"]` | Empty before-group |
| `"a ) b (c)"` | `["a ) b", "(c)"]` | Unbalanced close before open |

`batchSplitParens` composes `_peelLastParen` + `_assignInterpolatedTs`. A unit test here would catch regressions from any refactor of the split-parens logic — something your `mark.spec.js` tests only through the full import-then-mark flow.

### 3. Key Normalization & Hotkey Restriction Logic (high value)

| Function | Why unit test it |
|---|---|
| `keyStr(e)` | Modifier ordering, Space → "Space", `e.key.length === 1` → uppercase, Escape → "Esc" |
| `_normKey(k)` | Space, Escape, single char → upper, multi-char keys pass through |
| `isRestrictedForAll(ks)` | Comprehensive coverage of `RESTRICTED_ALL` membership |
| `isRestrictedForKey(ks, key)` | Mode-toggle restrictions for `toggle_mode` / `offset_mode_toggle` |
| `ALPHA_NUM_SPACE_RE` | Edge cases: `Shift+A`, `A`, `0`, `Space`, `F1` (should NOT match) |

**Why this matters:** Your `settings.spec.js` tests the capture UI flow, but doesn't exhaustively verify the restriction sets. If you ever add/remove a key from `RESTRICTED_ALL`, a unit test would confirm the change was intentional. The `ALPHA_NUM_SPACE_RE` regex is subtle — `Shift+F1` should NOT be blocked for mode toggles, but the regex currently doesn't test that boundary.

### 4. `mergeLrcMeta` and `ensureReTagDefault` (medium-high value)

These are pure string-manipulation functions that your Playwright specs only test through the full import flow. Specific test cases:

- `mergeLrcMeta`: LRC text with extra metadata fields not in defaults, empty values that should fall through to defaults, case-insensitive key matching
- `ensureReTagDefault`: `[re: existing]` → appends default URL, `[re: already-has-default]` → no-op, no `[re:]` tag at all → no-op

### 5. `cleanPaste` and `cleanGenius` (medium value)

- `cleanPaste`: Strips `[Section headers]` in paste context, preserves them in import context, normalizes 3-decimal timestamps
- `cleanGenius`: Detection of Genius-format content, boundary detection, "You might also like" filtering, section-header stripping, blank-line collapsing

Your `import-paste.spec.js` tests these through the UI, but `cleanGenius` has significant conditional logic that's hard to verify through snapshot comparison alone.

### 6. `collapseBlanks` (medium value)

Pure function, easy to test:
- Multiple consecutive blanks → single blank
- No blanks → no change
- Trailing blanks → single trailing blank
- Leading blanks → single leading blank

### 7. `_findNextTimestampMs` and `_assignInterpolatedTs` (medium value)

These implement the interpolation logic for inserted translation lines. Pure array/string processing. A unit test could verify the exact centisecond spacing formula (`nextMs - (count - k) * 10`).

## What I would NOT unit test

Functions that are tightly coupled to DOM state (`syncLine`, `adjustTs`, `renderMainLines`, `doUndo`, `doRedo`, `applyMode`, etc.) — these are already well-covered by Playwright and would require a heavy JSDOM or DOM shim to unit test, negating the value. The existing Playwright specs are the right tool for those.

## Practical approach

Since the app is a single-file HTML with all JS inline, extracting functions for unit testing requires either:

1. **`page.evaluate()`** — Run assertions inside the browser context via Playwright, accessing the in-scope functions directly. This keeps the app as-is but still gives you the precise diagnostics of unit tests. Example:
   ```js
   test('tsToMs parses standard timestamp', async ({ page }) => {
     const result = await page.evaluate(() => tsToMs('[01:23.45]'));
     expect(result).toBe(83450);
   });
   ```

2. **Extract a `logic.js` module** — Refactor pure functions into a separate `<script>` or module that both the app and a Node test runner import. More architectural change but enables `jest`/`vitest` without a browser.

Option (1) is the pragmatic win — it adds the diagnostic precision of unit tests with zero architectural change and fits naturally into your existing Playwright setup as a new `logic.spec.js` file.
